Agentic Self-Monitoring

How SafeMail watches its own health, and how an LLM agent diagnoses and fixes problems.

SafeMail runs a background loop that probes its own health, and when something breaks it hands the problem to a Claude agent. The agent investigates with a constrained set of tools, takes a safe automated fix when one applies, and otherwise escalates to a human — leaving a full audit trail behind. It does three things:

detect

Health probes catch broken polling, AI failures, queue backlogs, dead Redis, and revoked Gmail grants.

alert

A new problem opens an incident and emails ops, with the agent's diagnosis attached.

fix

The agent attempts bounded, reversible fixes — gated off by default until you trust it.

The big picture

A Celery beat task fires on a schedule (every MONITORING_INTERVAL_MINUTES, default 10). Each cycle runs every probe, reconciles findings against the open-incident table, and — only for a new problem — invokes the agent and sends an alert.

1 · Detection — the probes

Each probe is a small, isolated function in app/services/monitoring.py. It inspects one facet of the system and returns a Finding when something's wrong, or None when healthy. A probe that raises is caught and skipped, so one broken check can't blind the others.

ProbeFires when…Severity
check_redisRedis PING fails (broker + dedup set are down)critical
check_queue_backlogCelery queue depth exceeds the thresholdwarning
check_polling_livenessActive connections exist but no poll ran in the liveness window (beat/worker likely down)critical
check_stale_connectionsSome active connections haven't synced recently while others havewarning
check_task_failure_rateSustained spike in poll_connection / analyze_message failureswarning
check_terminal_api_failuresEmails dropped on non-retryable 4xx from Claude (bad/expired API key)critical
check_connections_in_errorGmail grants revoked/expired — child unmonitored until parent reconnectswarning

A Finding carries everything downstream needs — including a remediation_hint, plain-language guidance written for the agent (not the end user):

@dataclass
class Finding:
    check_name: str
    fingerprint: str        # stable dedup key, e.g. "polling_stalled"
    severity: str           # "warning" | "critical"
    title: str
    detail: str
    metrics: dict           # the numbers that tripped the probe
    remediation_hint: str   # what this means + which fixes to consider

2 · Incidents — dedup & lifecycle

Every distinct problem is one HealthIncident row, keyed by fingerprint. The cycle reconciles current findings against open incidents:

New

No open incident with this fingerprint → create one, run the agent, alert.

Recurring

Already open → bump times_seen, refresh metrics. No duplicate alert or re-run.

Gone

Open incident whose probe no longer fires → auto-resolve; send an "all clear" if it was alerted.

This is what keeps a flapping problem from paging you every 10 minutes: one incident, one alert, a running times_seen counter, and a clean auto-resolve.

3 · The agent — the core

This is the "agentic" part: app/services/remediation.py. Given a Finding, it runs a manual Claude tool-use loop — the model investigates with tools, decides on an action, and reports back. It uses claude-sonnet-4-6 (the model the product already uses for its classifier), runs synchronously inside the Celery worker, and never raises: any failure degrades to a recorded failed status so the incident still opens and alerts.

Why a manual loop and not the SDK tool-runner? The harness needs to do three things the auto-runner can't: gate mutating tools behind a config flag, cap how many fix actions run, and record every action for the audit trail. So we drive the loop ourselves.

The toolset

The agent can only do what a tool lets it — there is no shell, no SQL, no arbitrary writes. Tools come in three tiers:

always available · read-only
get_recent_task_logsinspect failures & error strings
get_connection_summarycounts by status + stale/errored lists
get_queue_depthcurrent Celery backlog
gated · mutating

Offered to the model only when AUTO_REMEDIATION_ENABLED=true.

requeue_gmail_pollingre-enqueue a full poll cycle (idempotent)
poll_connection_nownudge one active connection (idempotent)
always available
escalatedeclare the incident needs a human, with a reason and recommended action

The fix tools are deliberately the only two operations that are both idempotent (a Redis dedup set prevents reprocessing the same email) and safe to repeat. Anything that genuinely can't be auto-fixed — a bad API key, infrastructure down, a revoked OAuth grant only the parent can renew — has no fix tool, so the agent's only move there is escalate.

The agent loop, step by step

1

Assemble the prompt

System prompt casts the model as SafeMail's on-call SRE. The user turn contains the finding (title, detail, metrics, the probe's hint) and whether fix tools are available.

2

Call Claude with the tool set

Read tools + escalate always; the two fix tools only in auto mode. Each response's token usage is accumulated for cost reporting.

3

If it stopped to use tools, execute them

Read tools query the live DB / Redis. Fix tools and escalate are recorded in the actions list. Results go back to the model as tool_result blocks, and the loop continues.

4

Repeat until it stops calling tools

Bounded by MAX_AGENT_TURNS = 8. The model gathers what it needs, acts (or escalates), then ends its turn with a plain-text diagnosis.

5

Return a structured audit record

Diagnosis, every action with its result, the run mode, turn count, and token cost — written back onto the incident row.

StatusMeaning
attemptedA fix action ran. The next cycle's probe confirms whether it actually resolved.
escalatedNo safe fix applied — the agent flagged it for a human.
diagnosedAdvisory mode (or no action warranted): investigated and explained, took nothing.
failedThe agent itself errored (e.g. no API key). The incident still opens and alerts.

The safety model

The guiding principle is in the system prompt: a wrong automated action is worse than escalating. Several layers enforce it.

Off by default. AUTO_REMEDIATION_ENABLED=false means the agent runs in advisory mode — it investigates and recommends but is never even offered a mutating tool. You opt into automated fixes only once you trust them.
  • Allowlist, not capability. The agent can only call the tools it's given. No shell, no SQL, no free-form writes.
  • Bounded. Hard caps on agent turns (MAX_AGENT_TURNS = 8) and on mutating actions per incident (MAX_FIX_ACTIONS = 3), enforced by the loop regardless of what the model asks for.
  • Idempotent fixes only. The two fix tools are safe to repeat; the dedup set prevents double-processing email.
  • Guardrails inside tools. poll_connection_now refuses to touch a connection in error state — that needs the parent to re-grant OAuth, not a retry.
  • Fails safe. Any agent crash or API error is captured as failed; the cycle continues and the incident still alerts.
  • Untrusted output. The model-written diagnosis is HTML-escaped before it goes in an email (treated as untrusted text, per the codebase's OWASP-LLM05 stance).

4 · Alert & auto-resolve

A new incident emails ops via send_health_alert (SendGrid) — to OPS_ALERT_EMAIL, or every admin parent if that's unset. The email carries the severity, detail, the agent's diagnosis, the remediation status, and any actions taken. When the underlying probe stops firing, the incident auto-resolves and (if it had been alerted) an "all clear" follow-up goes out. Everything is also visible in the admin-only /monitoring console.

Observing it

Run the agent against your real DB + Redis on a synthetic finding and print exactly what it did:

cd backend && source .venv/bin/activate
AUTO_REMEDIATION_ENABLED=true python - <<'PY'
import json, redis
from app.config import get_settings
from app.database import SyncSessionLocal
from app.services.monitoring import Finding
from app.services.remediation import remediate

s = get_settings(); r = redis.from_url(s.redis_url)
finding = Finding(
    check_name="polling_liveness", fingerprint="polling_stalled",
    severity="critical", title="Gmail polling has stalled",
    detail="Active connections exist but no poll ran in the liveness window.",
    metrics={"active_connections": 3}, remediation_hint="Re-enqueue a poll cycle.")
with SyncSessionLocal() as db:
    print(json.dumps(remediate(finding, db=db, redis_client=r), indent=2, default=str))
PY

Or trigger a full cycle from the /monitoring console ("Run check now"), then expand the incident card to see the diagnosis and actions. The mocked test suite (pytest tests/test_monitoring.py) exercises the loop deterministically with no API cost.
